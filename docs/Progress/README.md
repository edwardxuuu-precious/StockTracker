# StockTracker 进度管理系统

## 📋 文档体系

本目录包含完整的项目进度管理文档,用于追踪项目目标、计划、进度和问题。

### 文档结构

```
docs/Progress/
├── Blueprint.md              # 项目蓝图 - 总目标和架构
├── Master_Plan.md            # 总体规划 - 任务拆解和测评标准
├── Master_Progress.md        # 总体进度 - 各目标完成情况
├── Div_Progress_Goal#x.md    # 分支进度 - 具体目标详细进度
├── Div_Todo_Goal#x.md        # 分支待办 - 具体目标待办事项
├── Current_Backlog_YYYY-MM-DD.md # 当前执行 backlog - 基于代码现状
└── Stop_Doing.md             # 问题边界 - 错误记录和约束
```

---

## 📚 文档说明

### 1. Blueprint.md - 项目蓝图

**用途:** 定义项目的总体目标和技术架构

**内容:**
- 项目概述和定位
- 总目标和核心指标
- 系统架构和技术栈
- 核心功能模块
- 数据模型设计
- 用户界面设计
- 部署架构
- 性能指标
- 安全设计
- 文档体系
- 成功标准
- 项目愿景

**更新频率:** 需求变更时更新

**阅读对象:**
- 项目所有成员
- 新加入的开发者
- 项目决策者

---

### 2. Master_Plan.md - 总体规划

**用途:** 将总目标拆解为可执行的具体任务

**内容:**
- 目标体系和分解树
- 各目标详细规划
- 任务拆解和步骤
- 完成标准定义
- 测评方式说明
- 时间规划安排

**更新频率:** 每周或阶段性更新

**阅读对象:**
- 开发团队
- 项目管理者
- 测试人员

**关键特点:**
- 每个任务都有明确的完成标准
- 每个任务都有具体的测评方式
- 每个任务都有预计工时

---

### 3. Master_Progress.md - 总体进度

**用途:** 追踪各目标的完成情况和整体进度

**内容:**
- 总体完成度统计
- 各目标完成情况
- 时间线和里程碑
- 进度趋势分析
- 工作量统计
- 关键成就记录
- 风险和阻塞
- 技术债务清单

**更新频率:** 每2-3天更新

**阅读对象:**
- 所有项目成员
- 项目管理者
- 利益相关者

**进度表示:**
```
✅ 已完成
🚧 进行中
⏳ 计划中
❌ 已取消
⚠️ 有风险
```

---

### 4. Div_Progress_Goal#x.md - 分支进度

**用途:** 详细记录某个具体目标的进度

**命名规则:**
- Goal#1 → `Div_Progress_Goal1.md`
- Goal#2 → `Div_Progress_Goal2.md`
- 以此类推

**内容:**
- 目标概述
- 子目标进度
- 任务完成情况
- 测评结果
- 关键成果
- 工作量统计
- 经验总结
- 验收标准

**更新频率:** 目标状态变更时更新

**示例:**
- [Div_Progress_Goal1.md](./Div_Progress_Goal1.md) - 基础设施建设 ✅

---

### 5. Div_Todo_Goal#x.md - 分支待办

**用途:** 管理某个具体目标的待办事项

**命名规则:**
- Goal#2 → `Div_Todo_Goal2.md`
- Goal#3 → `Div_Todo_Goal3.md`
- 以此类推

**内容:**
- 核心待办事项
- 优先级分类
- 时间规划
- 完成标准
- 不做事项清单
- 风险提示

**优先级分类:**
- P0 🔥🔥🔥 - 紧急且重要(立即完成)
- P1 🔥🔥 - 重要不紧急(计划完成)
- P2 🔥 - 紧急不重要(选择完成)
- P3 - 不紧急不重要(暂不实施)

**更新频率:** 每天或任务变更时

**示例:**
- [Div_Todo_Goal2.md](./Div_Todo_Goal2.md) - 投资组合管理待办 🚧

---

### 6. Stop_Doing.md - 问题边界

**用途:** 记录问题、错误和经验教训,作为开发约束

**内容:**
- 技术选择约束
- 架构设计约束
- 开发流程约束
- 代码质量约束
- 技术债务清单
- 经验总结
- 下次注意事项

**更新频率:** 遇到问题立即记录

**核心价值:**
- 避免重复犯错
- 积累经验教训
- 指导未来决策
- 降低试错成本

**记录格式:**
```markdown
### ❌ 不要做XXX

**问题描述:** 具体问题
**错误信息:** 错误提示
**根本原因:** 深层原因
**正确做法:** 解决方案
**预防措施:** 预防方法
**已解决:** ✅/⚠️/❌
```

---

### 7. Current_Backlog_YYYY-MM-DD.md - 当前执行清单

**用途:** 给出“现在就该做什么”的可执行 backlog,并和代码现状对齐

**内容:**
- 当前功能盘点结论
- 按 P0/P1/P2 排序的任务清单
- 每项任务的估时与验收标准
- 两周执行顺序建议

**更新频率:** 每次排期调整时更新

**示例:**
- [Current_Backlog_2026-02-08.md](./Current_Backlog_2026-02-08.md) - 当前执行版本

## 🎯 使用指南

### 新成员入职

**阅读顺序:**
1. Blueprint.md - 了解项目整体
2. Master_Plan.md - 了解目标和计划
3. Master_Progress.md - 了解当前进度
4. Current_Backlog_2026-02-08.md - 了解当前执行优先级
5. Stop_Doing.md - 了解约束和经验
6. 当前进行中的Div_Todo - 了解待办事项

**预计时间:** 1-2小时

---

### 日常开发

**每天开始:**
1. 查看 Master_Progress.md - 了解整体进度
2. 查看对应的 Div_Todo - 确认今日任务
3. 查看 Stop_Doing.md - 回顾注意事项

**开发过程:**
1. 按照 Div_Todo 中的优先级完成任务
2. 遇到问题立即记录到 Stop_Doing.md
3. 完成任务后更新进度

**每天结束:**
1. 更新 Div_Progress - 记录进度
2. 更新 Div_Todo - 勾选完成项
3. 记录明天的计划

---

### 阶段总结

**每周总结:**
1. 更新 Master_Progress.md
2. 回顾 Stop_Doing.md
3. 规划下周任务

**阶段完成:**
1. 完成 Div_Progress 总结
2. 更新 Master_Progress 里程碑
3. 归档经验教训

## 🔄 文档维护

### 维护原则

1. **及时更新**
   - 进度变化立即更新
   - 问题发现立即记录
   - 不要积累更新

2. **保持一致**
   - 各文档信息一致
   - 更新时同步相关文档
   - 避免信息冲突

3. **简洁清晰**
   - 使用简洁的语言
   - 重点突出
   - 格式统一

4. **可追溯**
   - 记录更新时间
   - 记录变更原因
   - 保留历史版本

### 更新流程

**进度更新:**
```
1. 完成一个任务
2. 更新 Div_Todo (勾选完成)
3. 更新 Div_Progress (记录进度)
4. 必要时更新 Master_Progress
```

**问题记录:**
```
1. 遇到问题
2. 记录到 Stop_Doing.md
3. 包含问题、原因、方案
4. 添加预防措施
```

**计划调整:**
```
1. 需求变更
2. 更新 Master_Plan
3. 调整 Div_Todo 优先级
4. 通知相关人员
```
